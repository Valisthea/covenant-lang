00 — Latest audit · 5 July 2026

OMEGA V6: 17 more findings, fixed the same day

Everything below this section documents the original OMEGA V4 audit before V0.7 GA — still accurate, still worth reading. But it is no longer the whole story. On 5 July 2026, a fresh breadth-first adversarial sweep across the compiler, stdlib, and produced-contract surfaces (the same "assume every guarantee is a lie" methodology described in Section 01) found 6 Critical, 6 High, and 5 Medium severity defects in the V0.9.2 compiler — the largest single-cycle count since the V0.6 launch audit. Every finding was empirically reproduced against source and compiled bytecode before being reported, and every fix shipped the same day with a new regression test. Full write-up: covenant-audits/audits/2026-07-05-omega-v6-covenant-v0.9.2.

Critical — two of the most basic constructs in the language

CRT-002
Critical Resolved

if without else silently deleted the success path

The dead-code eliminator treated a then-only if's implicit merge block as unreachable and deleted every statement after it — including the success path of the shipped 07_revert_paths.cov fixture. Source looked correct, the fixture had shipped, and nothing failed until this cycle inspected the emitted bytecode directly.

Fix: removed the Terminator::Unreachable guard causing the false-reachability classification in covenant-ir/src/builder.rs. Two new regression tests assert trailing statements after a then-only if survive DCE.

CRT-003
Critical Resolved

for each executed its body exactly once — it never actually iterated

The loop compiled to a header block that ran the body once and fell straight through to the merge block — no comparison, no back-edge, no bound loop variable. A ballot with 100 votes tallied exactly one of them, silently, with no diagnostic anywhere in the pipeline.

Fix: rewritten as a real block-parameter loop (header → body → merge, with a genuine comparison and back-edge jump). Landing this required implementing real list<Struct> storage end-to-end — append and list[idx].field read/write were also no-ops, now backed by keccak-derived dynamic-array slots (the Solidity convention). New end-to-end test: append three votes, sum with for each, verify each addressed slot independently.

CRT-004
Critical Resolved

only <builtin_predicate> guards compiled to unconditional true

Every BuiltinPredicate variant beyond owner/admin/deployer/an explicit address — around 16 variants, including first_time_caller and registered_key — codegenned to a bare PUSH1 0x01. Any contract gating an action behind one of these predicates silently authorized every caller.

Before — emitted bytecode
; only registered_key — NO CHECK
PUSH1 0x01   ; "Placeholder: return true"
; guard "passes" for every caller
After — hard compile error
error[E518]: `only registered_key`
has no real EVM authorization
check in this release — refusing
to compile a defeated guard

Fix: every unimplemented predicate now raises E518 and refuses to compile, matching the existing pattern for unlowered Shamir/VDF primitives (E516/E517) rather than shipping a guard that silently passes.

CRT-005 + HGH-030
Critical Resolved

Amnesia ceremony had zero threshold enforcement

A ceremony's entire purpose is threshold-of-N guardian consensus before destruction. finalize() trusted the (mocked) precompile's boolean with no on-chain corroboration that any guardian shares had actually been submitted — a ceremony could finalize with zero shares submitted. Separately, the ceremony synthesizer hardcoded a (3, 2) threshold regardless of the guardians:/threshold: values actually declared in the .cov source.

Fix: finalize() now asserts a real count of distinct submitters against the threshold read from the module's own metadata before it will even ask the precompile; submit_share dedupes by caller so one address can't unilaterally fabricate consensus. Three new regression tests cover zero-shares, one-of-two-submitters, and same-caller-twice.

CRT-006
Critical Resolved

Builtin identifiers silently shadowed same-named user fields

A user field or parameter named caller, now, block, msg, or current_block silently won or lost against the language-provided binding depending on scope-seeding order, with zero diagnostic either way — a name collision that could redirect a security-relevant read to the wrong value without any indication in source or compiler output.

Fix: the resolver now checks for an existing binding in the parent scope before seeding a builtin identifier, so a user declaration always wins deterministically and the collision is at least well-defined.

CRT-007
Critical Resolved

ERC-8231 PQ-key registry: ABI says dynamic, bytecode reads static

pq_key (used by the auto-synthesized registry construct's register(bytes) / key_of(...) returns bytes) declares ABI type bytes — dynamic, offset + length + data. Codegen could only read or return it as a single 32-byte word. Any spec-compliant caller (ethers.js, viem, a wallet, Etherscan) ABI-encoding a bytes argument places a dynamic-data offset at that word, not the key itself — the contract would silently register the constant 32 as the caller's "key," with no error, no revert.

Fix: the registry construct now hard-fails compilation with E505 until real dynamic-bytes storage and ABI encoding land — refusing to compile is strictly safer than shipping a compiler-caused key-corruption bug.

High — six issues, from a broken authorization surface to a process crash

IDFinding
HGH-026 A type-position identifier that resolves to something other than a struct/credential (a field, action, event…) silently lowered to Ty::Unknown with zero diagnostic. Now raises E231 whenever the identifier resolves to something real but non-type-shaped.
HGH-027 External-contract calls (IFoo.at(addr).method(args)) were never checked against the declared interface — wrong arity, wrong argument types, and typo'd method names all passed type-check silently. Now arity- and type-checked against the external contract block.
HGH-028 emit_external_call's non-view CALL path pushed addr before value — but CALL's real pop order needs value first, so every non-view external call attempted to "send" a fabricated huge value and failed with insufficient balance. A textbook case of Lesson 2 below — an asymmetric-operand test would have caught this instantly; the existing tests only checked opcode presence, never stack-slot ordering.
HGH-029 Every recursive-descent AST walker (parser, resolver, typechecker) recursed one native stack frame per nesting level with no depth counter. A few hundred bytes of nested parentheses, or a long chained arithmetic expression, overflowed the process stack — an uncatchable crash, not a normal compiler error — reachable from check, build, lint, test, and the LSP. Each stage now bounds its own recursion depth independently and raises a normal diagnostic instead.
HGH-031 No compiler-surfaced warning when a build routes an FHE/PQ/ZK primitive to a Mocked*.sol helper on a real public testnet — the "PLACEHOLDER, not for production secrets" labeling lived only in Solidity source comments a CLI user never sees. covenant build --target-chain=sepolia now prints an explicit warning naming the primitive and helper contract.

Medium

MED-001 — Privacy analyzer false positive: a legitimate list<Struct> write with a ciphertext<T> struct field was rejected as an E301 privacy violation, because the analyzer approximated the lvalue domain instead of resolving the precise field type. Now resolves the real field type recursively.

MED-003 — Non-constant dynamic-type returns: a view/reveal returning any non-compile-time-constant text/bytes/list value — including the language's own "Hello World" example, view read returns text { greeting } — silently ABI-encoded as a single raw word, with zero diagnostic. Now raises a warning naming the exact problem, deliberately not a hard error: an earlier attempt to hard-fail this broke seven existing test fixtures, including Hello World itself.

MED-004 — ListLength under-reporting: required no separate fix — resolved as a side effect of the CRT-003 real-storage rewrite above.

MED-005 — Mock precompile more permissive than the deployed testnet helper: the in-repo test double's ZK_VDF_VERIFY always returned true with no fail-mode at all, unlike every other verify-shaped mock precompile — local tests could never exercise "VDF proof rejected." Now gates on the same failure flag as the sibling ZK_VERIFY precompile.

MED-002 — Stale documentation: the project's own "known acceptable risks" ledger claimed no fuzz harness existed and that the parser had bounded recursion — neither was true before this cycle. A harness exists in-repo but has never been run; recursion is now genuinely bounded, thanks to HGH-029 above. Corrected in place.

Every fix in this section passed a full workspace rebuild, the full test suite (1050+ tests), and a clean clippy pass before moving to the next finding — and where a fix's first attempt caused new failures (HGH-027, MED-003), that correction is documented in the commit history, not hidden. All audits to date, OMEGA V4 through V6, are internal to Kairos Lab; an independent third-party audit remains the gate for V1.0 mainnet.

V0.9.2: the cycle that found every NFT was takeable

Critical — every auto-synthesized NFT could be taken by anyone

ERC-721 AUTH
Critical Resolved

transferFrom performed no caller authorization

The auto-synthesized transferFrom checked only ownerOf(token_id) == from — where from is an attacker-chosen parameter. It never loaded msg.sender at all. Any account could call transferFrom(victim, attacker, id) against any token the victim owned and seize it. Every NFT minted by a contract compiled with V0.9.0 or V0.9.1 was takeable by anyone who read the ABI. The audit fixture in the repo even documented the gate that was, in fact, absent from the emitted bytecode.

Fix: enforce OpenZeppelin _isAuthorized semantics — the caller must be the token owner, the per-token approved address, or an approved operator, else NotApprovedOrOwner. A shared emit_is_authorized helper also gates the newly synthesized burn. transferFrom now additionally reverts on to == address(0) (InvalidReceiver) and refuses a free-mint via from == address(0). (covenant-stdlib/src/erc721.rs.)

Honest scope: bytecode already deployed under V0.9.0/V0.9.1 is immutable — only new compiles get the gate. The M2 demonstration NFT on Sepolia therefore still carries the original permissive transferFrom, and is labelled as such on its showcase page.

High — four codegen defects that produced silently wrong contracts

IDFinding
KSR-CVN-029 Storage-slot collision — every map hashed keccak(key ‖ 0). Keyed map slots used the map's metadata-slot value (always 0 for a fresh map) instead of the slot number, so any two maps accessed with the same key aliased the same storage slot. In ERC-721, approve(spender, id) silently overwrote owners[id]; in the ERC-8231 registry, keys[caller] and registered[caller] clobbered each other. MapGet/MapSet/MapDelete now trace the base operand back to the field's slot number and hash keccak(key ‖ slot_number).
KSR-CVN-031 Checked arithmetic silently wrapped instead of reverting. AddChecked/SubChecked/MulChecked — the lowering of Covenant's +, -, * — emitted bare wrapping EVM ADD/SUB/MUL. So balance = balance - amount wrapped to ~type(uint256).max on underflow rather than reverting: the classic pre-0.8 Solidity balance exploit, reintroduced by codegen under a language that promised checked arithmetic. Now emits the overflow/underflow check and reverts.
KSR-CVN-030 Every external-contract call returned 0/default. After the STATICCALL/CALL, codegen zeroed mem[0x00] before reading it — wiping the return word the call had just written to retOffset=0. Every IFoo.at(addr).method() view silently returned zero. The post-call wipe is removed and the return word is guarded by RETURNDATASIZE, so an empty return still yields 0. Completes the M3 cross-contract milestone for statically-typed returns.
KSR-CVN-042 @slot(N) could silently alias reserved compiler slots. A user-specified @slot landing on DEPLOYER_SLOT (which backs only deployer) or REENTRANT_LOCK_SLOT (which backs @non_reentrant) hijacked the auth check or disabled the reentrancy guard, with no diagnostic. Now reported through the E423 slot-conflict path.

The pattern across all five is the one this page keeps returning to: the source was correct, the tests passed, and the bytecode did something else. Three of the four High findings were invisible to a test suite that asserted on opcode presence rather than on runtime values — the same blind spot Lesson 2 below describes, and the reason the V0.9.3 cycle above went looking in the emitted bytecode first.

01 — Methodology

OMEGA V4: assume every guarantee is a lie

OMEGA V4 is Kairos Lab's internal adversarial audit framework. It is not a checklist. It's a mindset: assume every guarantee the compiler claims to enforce is a lie, and try to prove it. The question is never "does the source look right?" — it's "does the emitted bytecode contain the bytes the source promised?"

  • Phase 1
    Reconnaissance Map the full attack surface. Parse every crate. Identify every place where source intent becomes bytecode — every opcode, every emit path, every optimization pass.
  • Phase 2
    Adversarial deep-dive For every guarantee the compiler claims, construct a source program that should work but the compiler silently breaks. Look at the actual bytecode. Verify the emitted code provides the guarantee.
  • Phase 3
    Reproducible PoCs For each finding, produce a minimal working example. Compile it. Simulate deployment. Show — with annotated bytecode — exactly how the bug manifests in production.
  • Phase 4
    Remediation Fix each finding with a regression test. The test alone, without the fix, must fail. The test with the fix must pass. No exceptions — every finding has a corresponding test that would have caught it.
  • Phase 5
    Re-audit After all fixes are merged, re-run every PoC. Verify no regressions. Look for new attack vectors that fixes might have introduced. Confirm the compiler is clean before GA.
02 — Critical Findings

Five failures that would have made every contract a honeypot

These findings share a pattern: the source looked correct, the tests passed, the contracts deployed — but the generated bytecode did not enforce the guarantee the source promised. Silent codegen failures are the worst class of bug.

KSR-CVN-011
Critical Resolved

IrGuard never codegenned — access control was fiction

Covenant's access control system lets developers write only(owner), when(condition), and given(proof) clauses on actions. These should emit a CALLER + EQ + ISZERO + JUMPI pattern targeting the revert path before any function body executes.

The IR builder accepted and type-checked the clause, but the emit_only_assert function was never called. Every deployed Covenant contract with an access control clause would silently accept calls from any address.

Before — emitted bytecode
; withdraw_all() — NO ACCESS CHECK
JUMPDEST
PUSH1 0x00
SLOAD         ; load balance
CALLER        ; push msg.sender
; (transfer — no guard before it)
PUSH1 0x00
PUSH1 0x00
SSTORE        ; clear balance
STOP
After — with guard
; withdraw_all() — GUARDED
JUMPDEST
CALLER        ; msg.sender
PUSH32 …      ; stored owner
SLOAD
EQ
ISZERO
PUSH2 revert  ; revert if ≠
JUMPI
; (transfer follows)

Fix: emit_only_assert in covenant-ir/src/builder.rs now emits LoadCaller → LoadPrincipal → Eq → Assert. The EVM backend lowers Assert to ISZERO + JUMPI → __revert__. Regression test in covenant-evm-backend/tests/guard_codegen.rs asserts every only() in source produces the CALLER + EQ + ISZERO + JUMPI pattern in the bytes.

KSR-CVN-012
Critical Resolved

Proxy initializer had no re-init guard — one transaction hijack

Covenant supports UUPS upgradeable proxy patterns. An initializer action is supposed to be callable exactly once. But the bytecode had no re-init guard — initialize() could be called repeatedly, overwriting owner and treasury with attacker-controlled addresses.

Fix: emit_initializer_guard wraps every initialize action with a slot-based flag. The slot is keccak256("covenant.proxy.initializer.<ModuleName>") — EIP-7201 namespaced, unique per module. The bytecode reads the slot, reverts if already set, then sets it before executing the body. Phase 3 is terminal — no re-entry possible.

KSR-CVN-013 + 014
Critical Resolved

Precompile call safety — PQ signature bypass via stale memory

Cryptographic precompiles (FHE ops, Dilithium verify, ZK proof check) are invoked via STATICCALL. Two separate failures combined into a full bypass of post-quantum authentication:

CVN-013: The STATICCALL success flag was popped and discarded. A precompile that reverted (invalid input, out of gas) was treated as successful.

CVN-014: Before STATICCALL, the return memory slot was not zeroed. A failed STATICCALL left stale memory that MLOAD read as a nonzero value — interpreted as "signature verified".

Combined exploit: An attacker submits a forged Dilithium signature. The precompile reverts. The success flag is dropped. The MLOAD reads stale memory containing a nonzero value from a prior operation. Authentication succeeds. One transaction, deterministic in deployments with prior operations.

; Three-layer defense added to every precompile call:
PUSH0 PUSH0 MSTORE ; 1. zero the return slot
(args setup)
STATICCALL
ISZERO ; 2. check success flag
PUSH2 revert_label
JUMPI
RETURNDATASIZE
PUSH1 32
EQ ; 3. check return size == 32
ISZERO
PUSH2 revert_label
JUMPI

Bytecode differential: runtime bytecode grew from 835 → 1336 bytes. The ISZERO opcode count went from 9 → 47. That's 38 new safety checks, one per precompile call site. CI now asserts ISZERO count as a lower bound.

KSR-CVN-001
Critical Resolved

Ceremony phase transitions unchecked — cryptographic amnesia bypass

Covenant implements the Amnesia Ceremony pattern (originally drafted in this repo as "ERC-8228" before that EIP number was officially assigned to the unrelated Styx Encrypted Token Standard; the Covenant ceremony is no longer attached to that EIP number). Ceremonies have a strict four-phase state machine: idle → gathering → finalized → destroyed. Only phase reads were checked; phase transitions had no guards. An attacker could call destroy() at phase 0, jump to phase 3, and have is_destroyed() return true without a key ever having existed.

Downstream integrations (GDPR compliance, regulatory data erasure, institutional key destruction proofs) rely on is_destroyed() being trustworthy. With transitions unchecked, the state machine was decorative.

Fix: Every ceremony function now checks its required precondition phase via emit_assert_phase_eq(expected_phase). Phases are enforced at bytecode level via SLOAD + EQ + ISZERO + JUMPI. Phase 3 is terminal — no function takes phase 3 as a valid input state.

04 — Medium · Low · Informational

23 further findings — the long tail of defense

Medium findings affected correctness and developer safety. Low and Informational findings strengthened tooling, diagnostics, and supply chain. All resolved before V0.7 GA.

KSR-CVN-022, 023 — Shamir / VDF hard-fail: Shamir Secret Sharing and Wesolowski VDF opcodes emitted broken stubs that silently succeeded without performing any cryptographic work. Rather than emit silent fiction, the compiler now hard-fails with E516/E517 for these opcodes until V0.8 chain support ships. "Deploy no code" beats "deploy broken code."

KSR-CVN-024, 025 — P4/P5 invariant detectors: Privacy invariants P4 (nonce reuse) and P5 (monotonicity) had no lint detectors in the security pass. Detectors added.

KSR-CVN-026 — FheBootstrap eliminated by DCE: Dead Code Elimination was removing FheBootstrap calls (which refresh FHE ciphertext noise budget) because they appeared to have no side effects. DCE now marks FHE refresh ops as side-effecting.

Storage / supply chain: @slot(N) collision detection wired (E423), annotation validation, PRECOMPILE_ABI_VERSION marker in bytecode, Cargo.lock pinning, per-chain version binding. Full list in covenant-audits.

05 — What the audit taught us

Four lessons that change how we write tests

  • Lesson 1
    Test the bytes, not the source For every safety guarantee your compiler claims, write a test that inspects the emitted bytecode and asserts the exact opcode pattern. only(owner) should produce CALLER + EQ + ISZERO + JUMPI. If it doesn't, the test fails — not because the source looked wrong, because the bytes are wrong.
  • Lesson 2
    Asymmetric tests catch operand ordering bugs Every test with symmetric values (Add(3,3), Eq(5,5)) would pass even if operands are reversed. We now test every binary op with asymmetric pairs (Sub(10,3) vs Sub(3,10), Lt(2,7) vs Lt(7,2)).
  • Lesson 3
    State machines must be enforced at the bytecode level Writing state = Phase2 in a function body doesn't protect you if the function can be called when state is Phase0. Every transition needs a precondition check compiled into bytecode — not documented in source, not enforced by convention.
  • Lesson 4
    Defense in depth for external calls STATICCALL has three failure modes: success flag, return data size mismatch, revert. Check all three. If any single check would have caught CVN-013 or CVN-014 alone, the exploit would have been impossible. That's what defense in depth means.
06 — Known limitations

What V0.9 does not yet support

Shipping broken stubs is worse than shipping nothing. These features are deliberately disabled until they can be implemented correctly.

⛔
Shamir Secret Sharing (E516)
Contracts using shamir_split, shamir_recover, or @shamir_threshold will not compile. Full implementation requires Aster Chain support (V0.8).
⛔
Wesolowski VDF (E517)
Contracts using vdf_eval or @vdf_locked will not compile. Same dependency on Aster SDK for chain-native VDF verification.
🔧
Aster Chain bytecode emission (V0.7 foundation only)
--target-chain aster compiles to a placeholder artifact (COV7\x01 magic + metadata). Full bytecode emission requires Aster SDK GA. V0.7 artifacts are not deployable on-chain.
🔧
External audits (deferred to V1.0)
OMEGA V4, V5, and V6 are all Kairos Lab self-audits. Independent third-party audits are the gate for V1.0 mainnet — not yet scheduled. Self-audit is not a substitute for independent verification, and OMEGA V6 finding 6 more Critical-severity defects in a compiler that had already passed three prior self-audit cycles is itself evidence of that, not a reason to relax the bar.
